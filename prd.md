# PRD — Auto-Balance AI 공개용 완성 명세

> 목적: 학생 캡스톤 제출본을 **공개 포트폴리오로서 "데모 가능한" 상태**로 끌어올린다.
> 작성: 2026-06-19 · 대상 저장소: https://github.com/blackdew/auto-balance-ai
> 범위 한정: **데모 수준 정리** — 학생 소스 로직은 보존하고, 보여주기·실행 경험만 완성한다.

---

## 1. 배경

`AI 게임 마스터(GM)`가 플레이어 실력을 학습해 다음 층 적 스탯을 자동 밸런싱하는 C++17 콘솔 턴제 RPG. KNN(k=5) 행동 예측 + 몬테카를로 시뮬(50회)로 목표 난이도(보스 승률 40%, 일반 층수 비례 HP 소모)를 데이터로 역설계하는 것이 핵심 기여다. 상세 구조는 `README.md` 참조.

이미 동작하는 완성된 게임이지만, **공개 저장소로서의 정리는 안 되어 있다**. 이 문서는 그 격차를 메우는 작업을 정의한다.

---

### 핵심 문제 — "AI가 안 보인다"

자동 밸런싱 튜너는 `std::async`로 **백그라운드에서 돌고 결과(적 스탯)만 다음 층에 조용히 적용**된다. 따라서 플레이 화면을 그대로 녹화하면 평범한 콘솔 RPG처럼 보이고, 이 프로젝트의 **진짜 성과(데이터 기반 난이도 역설계)가 데모에서 드러나지 않는다.** 공개 정리의 최우선 과제는 이 가시화다.

## 3. 목표 / 비목표

### 목표 (Goals)
- G1. 누구나 클론 후 **한 명령으로 빌드·실행**할 수 있다.
- G2. 저장소를 보는 사람이 **코드를 실행하지 않고도** 자동 밸런싱이 동작함을 확인할 수 있다 (데모 산출물).
- G3. 빌드가 깨지지 않음을 **CI가 지속적으로 증명**한다.
- G4. 공개 저장소로서 기본 요소(라이선스, 정돈된 README, 토픽)를 갖춘다.

### 비목표 (Non-Goals)
- N1. 학생 소스 **로직 변경/리팩토링 안 함** (실·시뮬 로직 중복 통합, 단위 테스트, 튜너 탐색 알고리즘 개선 등은 범위 외 — README "보완" 섹션으로 남김).
- N2. 밸런싱 품질의 정량 검증 리포트는 **이번 범위 아님** (향후 확장 후보).
- N3. 게임 신규 기능(콜드스타트 해결, 밸런싱 시각화 GUI 등) 추가 안 함.

## 4. 요구사항

### R1. 빌드 자동화
- 루트에 `Makefile` 추가. `make` → `src/turn_rpg` 생성, `make run`, `make clean`.
- `g++`/`clang++` 자동 감지, `-std=c++17 -O2 -Wall`.
- `vs_project/`의 깨진 경로 의존(README §실행방법 주석 참고)은 **건드리지 않고**, 표준 빌드 경로를 1순위로 명시.

### R2. 데모 모드 (additive — 로직 불변)
- `turn_rpg`에 **추가적인** 실행 경로 신설(예: `--demo` 인자 또는 별도 `demo_main`).
- 자동 플레이로 1→20층을 진행하며, **매 층 튜너의 결정을 출력**:
  - 시도한 후보 수, 채택된 적 스탯(atk/def/hp), 적 타입/메커니즘,
  - 목표값 대비 달성값 (보스: 목표 승률 40% vs 시뮬 달성 승률 / 일반: 목표 HP 소모 vs 달성).
- 기존 대화형 플레이는 **그대로 유지** (기본 동작 불변). 데모는 비대화형(입력 대기 없음)으로 끝까지 자동 실행.
- 제약: `ai.h`/`combat.h`의 밸런싱·전투 로직은 호출만 하고 수정하지 않는다. 출력/주행만 추가.

### R3. 데모 산출물 캡처
- R2 데모 실행을 **asciinema 캐스트 또는 애니메이션 GIF**로 캡처해 저장소에 포함.
- README 상단에 임베드 + 핵심 출력 발췌 텍스트 블록.

### R4. CI
- `.github/workflows/build.yml` — push/PR 시 Ubuntu에서 `g++ -std=c++17` 컴파일 + `--demo` 1회 스모크 실행(비대화형이므로 CI에서 완주 가능)으로 회귀 방지.
- README에 빌드 상태 배지.

### R5. 저장소 정리
- `LICENSE` 추가 (기본 제안: MIT — **학생/원작자 귀속 확인 필요**, §7 리스크 참조).
- README: 영문 1문단 요약 + 데모 GIF + 빌드 배지 보강.
- GitHub 저장소 토픽 설정(`cpp`, `game`, `knn`, `monte-carlo`, `auto-balancing`, `capstone`).

## 6. 작업 분해 (마일스톤)

1. **M1 — 빌드/저장소 기본기**: `Makefile`, `LICENSE`, README 배지 자리. (R1, R5 일부)
2. **M2 — 데모 모드**: `--demo` 비대화형 자동 주행 + 튜너 결정 출력. (R2) ← 핵심

## 7. 리스크 / 확인 필요

- **라이선스 귀속**: 원작자는 학생(원본 zip: `(완료)auto balance ai...`, Windows/VS 환경). 공개·라이선스 부여에 대한 **학생 동의 여부 확인 필요.** 미확인 시 `LICENSE` 보류 또는 원작자 명시 후 적용.
- **데모 모드 비결정성**: 튜너가 `mt19937` 난수를 쓰므로 데모 출력이 매 실행 달라진다. CI 스모크는 "완주 여부"만 검증(출력값 단언 금지)으로 안정화.
- **데모와 실전 로직 일치**: README §보완이 지적한 실·시뮬 로직 중복은 본 범위에서 해결하지 않음. 데모는 실전 경로를 그대로 호출하므로 추가 괴리는 만들지 않는다.

## 부록 — 채택 근거

대안 비교: (A) 순수 프레젠테이션 — AI가 데모에서 안 보여 메시지 약함. (B) **데모 모드로 AI 가시화 — 채택**. (C) 헤드리스 정량 검증 리포트 — 코드 품질 영역까지 확장되어 "데모 수준" 범위 초과. B는 단 하나의 치명적 데모 약점(AI 비가시성)을 로직 보존하에 해결한다.
